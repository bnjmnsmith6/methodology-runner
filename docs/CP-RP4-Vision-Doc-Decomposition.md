# Constellation Packet — RP-4: Vision Document Generator + Decomposition Engine
**Project:** GUPPI Vision Conversation Mode
**Tier:** 2
**Date:** 2026-03-25
**Dependencies:** RP-1 (Data Layer), RP-2 (Classifier), RP-3 (Conversation Manager)

---

## Context Block

RP-1 built the database. RP-2 built the classifier. RP-3 built the conversation engine. When the conversation is done, we have a completed Vision Session with messages, coverage state, and a summary. Now we need to turn that into two things:

1. **A Vision Document** — the canonical structured context for the entire project
2. **An RP decomposition** — the right number of RPs at the right tiers with the right dependencies

The Vision Document becomes the shared memory that every downstream agent references. The decomposition becomes the actual work structure. Both are generated by Claude Sonnet from the conversation output.

For fast-path requests, the Vision Document was already built by `buildFastPathVision()` in RP-3 and decomposition is trivially one RP. This RP handles the micro-vision and full-vision cases where conversation happened and real decomposition is needed.

**Tech Stack:** TypeScript, Anthropic API (Claude Sonnet), imports from RP-1/2/3, Supabase

---

## Implementation Spec

### What to Build

Two modules:
1. **Vision Document Generator:** Takes a completed vision session and produces a structured VisionDocument
2. **Decomposition Engine:** Takes a VisionDocument and proposes an RP graph with tiers, dependencies, and rich descriptions

### Build Order

**Step 1: Vision Document Builder**

Create `src/vision/buildVisionDoc.ts`:

```typescript
import { VisionDocument, VisionSession, VisionMessage, CoverageState } from '../types/vision';

export async function buildVisionDocument(
  session: VisionSession,
  messages: VisionMessage[],
  coverage: CoverageState
): Promise<VisionDocument>
```

Implementation:
- Call Claude Sonnet with the full conversation transcript and the coverage state
- Ask it to produce a structured JSON object matching the VisionDocument schema
- System prompt:

```
You are generating a Vision Document from a project intake conversation. 

Read the conversation transcript below. Extract and organize the information into the required JSON structure.

Rules:
1. Only include information that was explicitly stated or clearly implied in the conversation.
2. For fields where information was not discussed, use null or empty arrays.
3. Mark assumptions explicitly in the risk_register.assumptions array.
4. The one_sentence_brief should be crisp and actionable — not a repeat of the conversation.
5. success_criteria should be testable statements, not vague goals.
6. non_goals should capture anything the user explicitly excluded or that's clearly out of scope.
7. decomposition_hints should suggest natural work boundaries — don't force them.

Coverage state (what we know vs. assume vs. don't know):
{coverageJSON}

Conversation transcript:
{formattedTranscript}

Return ONLY a valid JSON object matching this schema. No markdown, no preamble.
{schemaDescription}
```

- Token budget: max_tokens 2000 (Vision Doc can be substantial)
- Parse the JSON response. If parsing fails, retry once with a reminder to return only JSON.
- Store the Vision Document in the vision_documents table via vision-repo
- Return the VisionDocument

**Step 3: Decomposition Engine**

Create `src/decompose/proposeRPs.ts`:

```typescript
import { VisionDocument, RPProposal } from '../types/vision';

export async function proposeRPDecomposition(
  visionDoc: VisionDocument
): Promise<{
  proposals: RPProposal[];
  explanation: string;
  suggestedBuildOrder: string[];
}>
```

Implementation:
- Call Claude Sonnet with the Vision Document
- Ask it to propose RP decomposition
- System prompt:

```
You are decomposing a project into Research Projects (RPs) — independent units of work that each go through research, review, spec, and build.

Read the Vision Document below and propose how to split this project.

Rules for splitting:
1. Split when work can be independently researched or validated
2. Split when work crosses a system boundary (data layer vs UI vs auth vs integrations)
3. Split when work has a different risk profile
4. Split when work blocks other work through an interface or contract
5. DO NOT split when tasks are tiny and tightly coupled
6. DO NOT split when the same acceptance test governs all of them
7. Default to FEWER RPs. Only split when there's a real boundary.
8. Soft cap: 6 RPs maximum unless the project truly warrants more.

For each RP, provide:
- title: clear, specific name
- description: 2-4 sentences with rich context pulled from the Vision Document. NOT just the user's original words — add relevant constraints, acceptance criteria, and context.
- tier: 1 (simple/bounded), 2 (moderate complexity), or 3 (architecture/security/data-heavy)
- dependencies: which other RPs must complete first, and why
- rationale: one sentence explaining why this is a separate RP
- pbca_focus: 2-3 specific research questions for the PBCA agent

Tier assignment:
- Tier 1: Bounded, reversible, low-risk. Single file or small component. Can be accepted by direct output inspection.
- Tier 2: Moderate ambiguity or integration. Multiple files/components. Some product judgment required.
- Tier 3: Architecture-heavy. Security/privacy/data/permissions. Significant reversibility cost. Needs full research + red-team.

Also provide:
- explanation: 2-3 sentences explaining your decomposition rationale to the user in plain English
- suggestedBuildOrder: array of RP titles in the order they should be built

Vision Document:
{visionDocJSON}

Return ONLY a valid JSON object:
{
  "proposals": [...],
  "explanation": "...",
  "suggestedBuildOrder": [...]
}
```

- Token budget: max_tokens 2000
- Parse JSON response
- Validate: each proposal has required fields, tiers are 1-3, dependencies reference valid titles
- If only one RP makes sense, that's fine — return one proposal

**Step 4: Tier Assigner (validation/override)**

Create `src/decompose/assignTier.ts`:

```typescript
import { RPProposal, VisionDocument } from '../types/vision';

// Validate and potentially override the LLM's tier assignments
export function validateTierAssignments(
  proposals: RPProposal[],
  visionDoc: VisionDocument
): RPProposal[]
```

Rules:
- If any proposal mentions auth/security/permissions AND tier < 3, upgrade to 3
- If any proposal mentions data migration/schema AND tier < 2, upgrade to 2
- If any proposal has 0 dependencies and description suggests single file, cap at tier 1
- If the Vision Document classification.complexity is 'simple', cap all tiers at 2
- Return the proposals with validated tiers

This is a pure function, no LLM calls.

## Constraints (What NOT to Do)

- DO NOT create actual projects or RPs in the projects/rps tables — that's RP-6
- DO NOT modify existing chat tools — that's RP-6
- DO NOT modify downstream PBCA/review/spec/build prompts — that's RP-5
- DO NOT make more than 2 LLM calls per decomposition (one for Vision Doc, one for RP proposals)
- DO NOT allow more than 6 RPs without explicit justification
- DO NOT generate vague RP descriptions — each one must have specific context from the Vision Document
- Use Claude Sonnet (claude-sonnet-4-20250514), same Anthropic client as Claude Brain adapter

---

## Stop and Ask List

1. **RP-3 not complete.** If conversation manager files don't exist, stop.
2. **Anthropic client.** Reuse the existing one from Claude Brain adapter.
3. **JSON parsing failures.** If Claude returns invalid JSON on the first try, retry once. If it fails twice, return a single-RP fallback with the raw Vision Document as description.
